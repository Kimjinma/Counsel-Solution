# 종합 상담 솔루션 서비스 (Counsel-Solution)

> **대학 내 상담 프로세스(신청-관리-후속 조치)를 웹 기반 플랫폼으로 통합한 종합 솔루션입니다.**
> 기존의 파편화된 수기 관리 시스템을 개선하여 학생과 상담 기관 모두의 편의성을 높이기 위해 기획되었습니다.

## 프로젝트 정보
* **진행 기간:** 2024.11.01 ~ 2025.01.15
* **참여 인원:** 팀 프로젝트
* **본인 담당:** 인증(Auth) · 마이페이지(학생/상담사) · CS(만족도 조사) 후속 관리 시스템 전담
* **GitHub:** [https://github.com/Kimjinma/Counsel-Solution](https://github.com/Kimjinma/Counsel-Solution)

## Skills & Tech Stack
* **Backend:** `Java 17`, `Spring Boot`, `Spring Data JPA`, `Spring Security`
* **Database:** `MySQL`
* **Template Engine:** `Mustache`, `Thymeleaf`

---

## 나의 역할 및 주요 개발 기능 (My Contribution)

**1. 독립적인 인증 레이어 구축 및 페이지 접근 제어**
* 마이페이지(조회/수정) 기능과 만족도 조사 비즈니스 로직을 단독 테스트하기 위해, **Spring Security `SecurityFilterChain` 기반의 폼 로그인 인증 설정과 `UserDetailsService` 커스텀 구현을 주도적으로 구축**
* `@AuthenticationPrincipal`을 활용한 세션 기반의 사용자 식별 패턴을 수립하고, URL별 역할 기반 접근 제어(ADMIN/STUDENT/COUNSELOR)를 세분화하여 총 20개 이상의 인가 규칙 적용

**2. 학생·상담사 마이페이지 및 예약 이력 관리**
* 학생·상담사 두 역할에 대해 각각 독립된 마이페이지(개인정보 조회/수정)를 구현하고, 비밀번호·이메일·주소 등을 `@Transactional`로 원자적 업데이트
* 학생 마이페이지에서 상담·프로그램 예약 상태(승인/대기)를 확인할 수 있는 이력 조회 시스템 구축
* 상담사 마이페이지에서 담당 상담 목록 관리, 예약 승인/대기 상태 변경, 상담 기록 작성 및 REST API 기반 비동기 조회 기능 구현
* 학생-상담사-프로그램 간의 복잡한 외래키 의존성을 JPA `@ManyToOne` 단방향 연관관계 매핑(총 10개)으로 정리하여 객체 그래프 탐색 기반의 조회 로직 구현

**3. CS(만족도 조사) 후속 관리 시스템**
* 상담 완료 후 다중 질문(객관식 3문항 + 주관식 3문항) 만족도 조사 제출 API 설계 및 구현
* 상담 완료 여부 검증, `CS_YN` 플래그 기반 중복 제출 방지, UUID 기반 고유 응답 식별자 부여 등 데이터 무결성 보장 로직 적용

## 트러블 슈팅 (Troubleshooting)

<details>
<summary><b>1. 다중 질문 CS(만족도) 데이터 유실 방지</b></summary>
<div markdown="1">

* **문제:** 상담 완료 후 다중 질문에 대한 만족도(주관식/객관식)를 DB에 저장할 때, 필드 매핑 오류로 인해 마지막 입력 값만 덮어씌워져 데이터가 유실되는 버그 발생.
* **해결:** 항목 단위 Entity/컬럼으로 완벽히 분리 저장하도록 데이터베이스 모델 및 서비스 로직을 재설계.
* **결과:** 데이터 무결성 확보 및 향후 질문 추가 시에도 확장이 용이한 저장 구조 확립.
</div>
</details>

<details>
<summary><b>2. 민감 정보 하드코딩 개선</b></summary>
<div markdown="1">

* **문제:** 초기 개발 시 DB 접속 정보(비밀번호 등)가 소스 코드에 직접 하드코딩되어 보안 취약점이 존재함.
* **해결:** DB 접속 정보를 시스템 환경 변수(`DB_USERNAME`, `DB_PASSWORD`)를 우선 참조하도록 `DataSource` 설정을 분리하여 보안을 강화.
* **결과:** 소스 코드에서 민감 정보를 분리하고, 운영 환경별로 설정을 유연하게 관리할 수 있는 구조 확보.
</div>
</details>

<details>
<summary><b>3. ERD 선설계로 연관관계 혼선 정리</b></summary>
<div markdown="1">

* **문제:** ERD가 확정되지 않은 상태에서 엔티티부터 작성하여 FK(외래키) 설정과 연관관계 매핑에 혼선이 발생함.
* **해결:** 구현을 잠시 멈추고 `USER_INFO`를 중심으로 부분 ERD를 먼저 도식화하여 시스템 아키텍처를 시각적으로 구체화함.
* **결과:** 설계 확정 후 JPA 단방향 매핑을 한층 더 안정적으로 적용하며 객체 참조의 복잡성을 낮출 수 있었음.
</div>
</details>

## 프로젝트 회고 (Retrospective)

**"단일 책임 원칙(SRP)과 모듈 파편화"에서 얻은 교훈**
> 개발 진행 중 타 파트(통합 로그인 등)의 코드를 제 브랜치로 가져오는 과정에서 잦은 Merge Conflict가 발생했습니다. 기능 지연을 막기 위해 제가 담당한 마이페이지 및 CS 모듈 테스트용 자체 시큐리티 인증 환경을 단독 구축하여 문제를 돌파했지만, 결과적으로 레포지토리가 두 개의 폴더 기반으로 파편화되는 아키텍처적 부채(Anti-Pattern)를 경험했습니다. 이 뼈아픈 경험을 통해 도메인 단위의 명확한 서비스 분리, 협업을 위한 올바른 Git Flow 브랜치 전략, 그리고 공통 모듈 라이브러리화의 중요성을 실무 레벨에서 깊이 체감할 수 있었습니다.

## 로컬 환경 테스트 및 가동 방법

해당 프로젝트의 레포지토리는 과거 팀 협업 이슈로 인해 분리 구성되었으나, 현재는 본인이 주도적으로 설계한 마이페이지 및 핵심 로직 테스트를 위해 `project/project` 폴더를 기준으로 단일화되어 있습니다.
로컬 구동 시 MySQL DB 접속 정보를 환경 변수(`DB_USERNAME`, `DB_PASSWORD`)로 설정해야 합니다.

**1. 프로젝트 환경 빌드**
```bash
./gradlew clean build -x test
```

**2. 애플리케이션 실행**
```bash
./gradlew bootRun
```
